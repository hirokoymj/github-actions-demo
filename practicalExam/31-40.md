**Q31**: What jobs.job_id.if conditional will make sure that job production-deploy is triggered only on my-org/my-repo repository? (Select two.)

```
  jobs:
    production-deploy:
      if: <CONDITION>
      runs-on: ubuntu-latest
      steps:
        ...
```

- `if: ${{ github.repository == 'my-org/my-repo' }}` ✅
- `if: ${{ github.organization == 'my-org' && github.repository == 'my-repo' }}` ❌

💡 https://docs.github.com/en/actions/learn-github-actions/contexts#github-context

✅ Correct Answers (two):

1. if: `${{ github.repository == 'my-org/my-repo' }}`
2. if: `github.repository == 'my-org/my-repo'`

- These two are actually doing the exact same thing — GitHub Actions allows you to write if conditions with or without the `${{ }}` expression syntax:
- ❌ github.organization does not exist — it's a made-up context variable
- ❌ github.org does not exist either — also made-up

**Q33:** Is this statement true? Not all steps run actions, but all actions run as a step

- True
- False

💡 https://docs.github.com/en/actions/using-workflows/workflow-syntax-for-github-actions#jobsjob_idsteps

✅ Correct Answer: True

```yaml
steps:
  # This step runs an ACTION
  - name: Checkout code

  # This step runs a SHELL COMMAND (not an action)
  - name: Print message
    run: echo "Hello World"
```

- "Not all steps run actions" ✅
  - A step can either run an **action** OR a **shell command** directly using `run`:
- "All actions run as a step" ✅

- Reference the main branch❌
- Reference the commit SHA ✅

💡 https://docs.github.com/en/actions/using-workflows/workflow-syntax-for-github-actions#example-using-versioned-actions


**Q35:** To prevent a job from failure when one of the steps fails you can include:

- continue-on-error flag in the failing step

```yaml
steps:
  - uses: my-org/failing-action@v1
    continue-on-error: true
```

- ignore-error flag in the failing step

```yaml
steps:
  - uses: my-org/failing-action@v1

```

- failure() conditional in the failing step

```yaml
steps:
  - uses: my-org/failing-action@v1
    if: failure()
```

- always() conditional in the failing step

```yaml
steps:
  - uses: my-org/failing-action@v1
    if: always()
```

✅ Correct Answer: continue-on-error: true

- ignore-error: true - ❌ Does not exist in GitHub Actions
- if: failure() - ❌ This means "only run this step IF a previous step failed"
- if: always() - ❌ This means "always run this step regardless"

**Q36:** You defined a matrix job example_matrix. How can limit the matrix to run a maximum of 2 jobs at a time?

```yaml
jobs:
  example_matrix:
    strategy:
      matrix:
        version: [10, 12, 14]
        os: [ubuntu-latest, windows-latest]
```

- Set jobs.example_matrix.strategy.concurrency to 2
- Set jobs.example_matrix.strategy.max-parallel to 2

- 💡 https://docs.github.com/en/actions/using-workflows/workflow-syntax-for-github-actions#jobsjob_idstrategymax-parallel

✅ Correct Answer:

- Set `jobs.example_matrix.strategy.max-parallel` to 2 is the correct answer.

```yaml
jobs:
  example_matrix:
    strategy:
      max-parallel: 2 ✅
      matrix:
        version: [10, 12, 14]
        os: [ubuntu-latest, windows-latest]
```

- This tells GitHub Actions to run at most 2 matrix jobs simultaneously, even if more runners are available.

**Q37**: Which of these is a proper way of setting an output parameter PET with a value of DOG in a step.

- gh set-output "PET=DOG"
- echo "DOG=PET" >> "$GITHUB_OUTPUT"
- echo "PET=DOG" >> "$GITHUB_OUTPUT" ✅
- gh set-output "DOG=PET"

💡 https://docs.github.com/en/actions/using-workflows/workflow-commands-for-github-actions#setting-an-output-parameter

✅ Correct Answer:

- echo `"PET=DOG" >> "$GITHUB_OUTPUT"`

```yaml
steps:
  - name: Set output
    id: my-step
    run: echo "PET=DOG" >> "$GITHUB_OUTPUT"

  - name: Use output
    run: echo "The pet is ${{ steps.my-step.outputs.PET }}"
```

**Q38:** Which of these is a way of using action_state in step_two?

```yaml
  steps:
    - name: Set the value
      id: step_one
      run: |
        echo "action_state=yellow" >> "$GITHUB_ENV"
    - name: Use the value
      id: step_two
      run: ?
```

- run: echo "$steps.step_one.outputs.action_state"
- run: echo "${{ action_state }}"
- run: echo "$action_state" ✅
- run: echo "${{ steps.step_one.outputs.action_state }}" ❌

💡 https://docs.github.com/en/actions/using-workflows/workflow-commands-for-github-actions#example-of-writing-an-environment-variable-to-github_env

✅ run: echo "$action_state"

- `$GITHUB_ENV` vs `$GITHUB_OUTPUT` are different!

| Method         | How to set                            | How to read                       |
| -------------- | ------------------------------------- | --------------------------------- |
| $GITHUB_ENV    | echo "NAME=value" >> "$GITHUB_ENV"    | $NAME (plain env variable)        |
| $GITHUB_OUTPUT | echo "NAME=value" >> "$GITHUB_OUTPUT" | ${{ steps.step_id.outputs.NAME }} |

**Q40:** In the following example, workflow A passes all of its secrets to workflow B, by using the inherit keyword. Then workflow B calls workflow C. Which statement regarding secrets is true for that example?

```yaml
jobs:
  workflowA-calls-workflowB:
    uses: octo-org/example-repo/.github/workflows/B.yml@main
    secrets: inherit
```

```yaml
jobs:
  workflowB-calls-workflowC:
    uses: different-org/example-repo/.github/workflows/C.yml@main
```

- All secrets available to workflow A will be also available to workflow B, but not to workflow C ✅
- All secrets available to workflow A will be also available to workflow B and workflow C
- Only repository and environment secrets available to workflow A will be available to workflow B, but not to workflow C. Organization scoped secrets cannot be inherited
- All secrets from octo-org organization and octo-org/example-repo repository will be available to workflow B, but not to workflow C

💡 https://docs.github.com/en/actions/using-workflows/reusing-workflows#passing-secrets-to-nested-workflows

✅ Correct Answer:

- All secrets available to workflow A will be also available to workflow B, but not to workflow C

```yaml
# Workflow A → Workflow B  (uses inherit ✅)
jobs:
  workflowA-calls-workflowB:
    uses: octo-org/example-repo/.github/workflows/B.yml@main
    secrets: inherit   # ← explicitly passes ALL secrets to B

# Workflow B → Workflow C  (NO secrets passed ❌)
jobs:
  workflowB-calls-workflowC:
    uses: different-org/example-repo/.github/workflows/C.yml@main
    # no secrets: inherit here!
```

- Workflow A → Workflow B (✅ inherit all) → Workflow C (❌ no inherit - get NO secrets)
