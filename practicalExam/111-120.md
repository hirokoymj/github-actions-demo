## Q111: In a workflow with multiple jobs the default behavior is:

- All jobs run in parallel
- Jobs run in sequence
- Jobs run based on the order they are defined in the workflow file
- Only the first job runs, others require manual approval

💡 https://docs.github.com/en/actions/using-workflows/about-workflows#creating-dependent-jobs

✅ Correct answer:

```
- All jobs run in parallel ✅
- Jobs run in sequence
- Jobs run based on the order they are defined in the workflow file
- Only the first job runs, others require manual approval
```

💡 \*\*Why this is correct

- By default, GitHub Actions runs all jobs at the same time

```yaml
jobs:
  build:
    runs-on: ubuntu-latest

  test:
    runs-on: ubuntu-latest
👉 build and test run in parallel
```

## Q112: What level of access is required on a GitHub repository in order to delete log files from workflow runs?

- write
- read
- admin
- owner

💡 https://docs.github.com/en/actions/how-tos/monitor-workflows/use-workflow-run-logs

✅ Correct answer:

```
- write ✅
- read
- admin❌
- owner
```

- GitHub has 5 repository access levels:
- Read → Triage → Write → Maintain → Admin

| Access Level | What you can do with Actions                         |
| ------------ | ---------------------------------------------------- |
| **Read**     | View workflows, view logs                            |
| **Triage**   | Re-run workflows                                     |
| **Write**    | Delete log files, cancel runs, trigger workflows     |
| **Maintain** | Manage repo settings (limited)                       |
| **Admin**    | Manage secrets, manage environments, disable Actions |

So the pattern to remember for the exam is:

- **Read** = view only
- **Write** = interact and modify (including deleting logs)
- **Admin** = manage secrets, permissions, and settings

## Q118: How can you skip the following workflow run when you commit or create a PR?

```yaml
name: Build
on: [push, pull_request]

jobs:
  build:
    runs-on: ubuntu-latest
    name: Extract artifact version
```

- By including any one of the following keywords in the commit message or in the title of the pull-request✅
  💡 hhttps://docs.github.com/en/actions/how-tos/manage-workflow-runs/skip-workflow-runs

```
[skip ci]
[ci skip]
[no ci]
[skip actions]
[actions skip]
```

- Provide SKIP_WORKFLOW in the commit message
- The above workflow will run in every event of push or pull request in every case❌

## Q120: What is the correct syntax for specifying a cleanup script in a container action?

```yaml
#1
 runs:
   using: 'docker'
   image: 'Dockerfile'
   entrypoint: 'entrypoint.sh'
   after-entrypoint: 'cleanup.sh'
#2
 runs:
   using: 'docker'
   image: 'Dockerfile'
   entrypoint: 'entrypoint.sh'
   after: 'cleanup.sh'
#3
 runs:
   using: 'docker'
   image: 'Dockerfile'
   entrypoint: 'entrypoint.sh'
   cleanup: 'cleanup.sh'
#4
 runs:
   using: 'docker'
   image: 'Dockerfile'
   entrypoint: 'entrypoint.sh'
   post: 'cleanup.sh'
#5
 runs:
   using: 'docker'
   image: 'Dockerfile'
   entrypoint: 'entrypoint.sh'
   post-entrypoint: 'cleanup.sh'

```

💡 https://docs.github.com/en/actions/creating-actions/metadata-syntax-for-github-actions#runspost-entrypoint

✅ Correct answer:

```yaml
#5
 runs:
   using: 'docker'
   image: 'Dockerfile'
   entrypoint: 'entrypoint.sh'
   post-entrypoint: 'cleanup.sh' ✅
```

**💡 Why this is correct**

```yaml
# 1️⃣ JavaScript Action
runs:
  using: 'node20'
  main: 'index.js'

# 2️⃣ Docker Action
runs:
  using: 'docker'
  image: 'Dockerfile'
  entrypoint: 'entrypoint.sh'
  post-entrypoint: 'cleanup.sh'  # cleanup

# 3️⃣ Composite Action
runs:
  using: 'composite'
  steps:
    - run: echo "Hello"
```

| Type       | Runtime    | Cleanup keyword    | Use case                 |
| ---------- | ---------- | ------------------ | ------------------------ |
| JavaScript | Node.js    | `post:`            | API logic, tools         |
| Docker     | Container  | `post-entrypoint:` | Full environment control |
| Composite  | YAML steps | N/A                | Reusable step grouping   |

🚨 **Exam trick:** JS = `post` / Docker = `post-entrypoint` / Composite = no cleanup keyword
