Q111: In a workflow with multiple jobs the default behavior is:

- All jobs run in parallel ✅
- Jobs run in sequence
- Jobs run based on the order they are defined in the workflow file
- Only the first job runs, others require manual approval

💡 https://docs.github.com/en/actions/using-workflows/about-workflows#creating-dependent-jobs

✅ Correct answer: 👉 All jobs run in parallel

- 💡 **Why this is correct**
- By default, GitHub Actions runs all jobs at the same time

```yaml
jobs:
  build:
    runs-on: ubuntu-latest

  test:
    runs-on: ubuntu-latest
👉 build and test run in parallel
```

### Q120: What is the correct syntax for specifying a cleanup script in a container action?

```yaml
#1
 runs:
   using: 'docker'
   image: 'Dockerfile'
   entrypoint: 'entrypoint.sh'
   after-entrypoint: 'cleanup.sh'
#2
 runs:
   using: 'docker'
   image: 'Dockerfile'
   entrypoint: 'entrypoint.sh'
   after: 'cleanup.sh'
#3
 runs:
   using: 'docker'
   image: 'Dockerfile'
   entrypoint: 'entrypoint.sh'
   cleanup: 'cleanup.sh'
#4
 runs:
   using: 'docker'
   image: 'Dockerfile'
   entrypoint: 'entrypoint.sh'
   post: 'cleanup.sh'
#5
 runs:
   using: 'docker'
   image: 'Dockerfile'
   entrypoint: 'entrypoint.sh'
   post-entrypoint: 'cleanup.sh' ✅

```

💡 https://docs.github.com/en/actions/creating-actions/metadata-syntax-for-github-actions#runspost-entrypoint

✅ Correct answer:

```yaml
runs:
  using: 'docker'
  image: 'Dockerfile'
  entrypoint: 'entrypoint.sh'
  post-entrypoint: 'cleanup.sh'
```

💡[doc](https://docs.github.com/en/actions/creating-actions/metadata-syntax-for-github-actions#runspost-entrypoint)

- 💡 **Why this is correct**
- In Docker container actions, GitHub Actions supports a special lifecycle hook:
  - `entrypoint` → main execution script
  - `post-entrypoint` → cleanup script after execution
- 👉 This is the only valid cleanup mechanism in this context.

#### 🧠 GitHub Actions Action Types - 3

#### 1️⃣ JavaScript Action (most common)

```yaml
runs:
  using: 'node20'
  main: 'dist/index.js'
```

- 💡 Key points: Runs in Node.js runtime

```
action/
├── action.yml
└── index.js
```

#### 2️⃣ Docker Action (container-based)

```yaml
runs:
  using: 'docker'
  image: 'Dockerfile'
  entrypoint: 'entrypoint.sh'
  post-entrypoint: 'cleanup.sh'
```

💡 Key points:
Runs inside a Docker container
Good for:
complex environments
system-level dependencies

🧩 Lifecycle

```
entrypoint → main execution
post-entrypoint → cleanup
```

🧠 Key takeaway

- `Docker action = full environment control`

3️⃣ Composite Action (workflow-like steps)

```yaml
runs:
  using: 'composite'
  steps:
    - run: echo "Hello"
```

#### 💡 Key points:

- Combines multiple steps
- No programming language required
- Best for:
  - reusable workflows
  - grouping commands

🧩 Example

```yaml
runs:
  using: 'composite'
  steps:
    - run: npm install
    - run: npm test
```

```
🚨 Exam Memory Trick
JS = logic
Docker = environment
Composite = steps
```

| Type       | Runtime    | Use case           |
| ---------- | ---------- | ------------------ |
| JavaScript | Node.js    | API logic, tools   |
| Docker     | Container  | Full environment   |
| Composite  | YAML steps | Reusable workflows |
