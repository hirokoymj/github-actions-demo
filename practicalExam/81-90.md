Q81: How can you reuse a defined workflow in multiple repositories? (Choose two.)

- By creating a reusable action
- By copying the workflow file to each repository
- By using workflow templates✅
- By defining the workflow in a central repository✅

💡 https://docs.github.com/en/actions/using-workflows/creating-starter-workflows-for-your-organization

- **Workflow Templates (Starter Workflows)**
- `.github/workflow-templates/node-ci.yml`

```yaml
name: Node.js CI

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Set up Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'
      - run: npm ci
      - run: npm test
```

```bash
org-name/.github                     ← template lives here
    └── workflow-templates/
            ├── node-ci.yml
            └── node-ci.properties.json

any-other-repo/.github/workflows/    ← copy lands here after selection
            └── node-ci.yml
```

Q83: What does the needs keyword do in a GitHub Actions workflow?

- Specifies the dependencies of a job ✅
- Defines environment variables
- Sets up the environment
- Triggers a job based on an event

💡 https://docs.github.com/en/actions/using-workflows/workflow-syntax-for-github-actions#jobsjob_idneeds

- Correct! The needs keyword creates a dependency between jobs

```yaml
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - run: echo "Building..."

  test:
    runs-on: ubuntu-latest
    needs: build # waits for build to complete first
    steps:
      - run: echo "Testing..."

  deploy:
    runs-on: ubuntu-latest
    needs: [build, test] # waits for BOTH build and test
    steps:
      - run: echo "Deploying..."
```

- build → test → deploy

Q84: Which keyword allows you to define environment variables in a GitHub Actions workflow?

- env✅
- config
- secrets
- vars

💡 https://docs.github.com/en/actions/using-workflows/workflow-syntax-for-github-actions#jobsjob_idenv

- The env keyword defines environment variables and can be set at three different levels:
- **Workflow level** — available to all jobs and steps:

```yaml
env:
  NODE_ENV: production
  API_URL: https://api.example.com

jobs:
  build: ...
```

- **Job level** — available to all steps within that job only:

```yaml
jobs:
  build:
    runs-on: ubuntu-latest
    env:
      BUILD_MODE: release
    steps: ...
```

- **Step level** — available to that single step only:

```yaml
steps:
  - name: Run tests
    env:
      TEST_TIMEOUT: 30
    run: npm test
```

- secrets — secrets are a separate concept accessed via ${{ secrets.MY_SECRET }}; you can assign a secret to an env var but secrets itself is not the keyword for defining env vars

Q85: What is the purpose of the `with` keyword in a GitHub Actions workflow?

- To specify input parameters for an action✅
- To trigger another workflow❌
- To define environment variables❌
- To set up dependencies❌

💡 https://docs.github.com/en/actions/using-workflows/workflow-syntax-for-github-actions#jobsjob_idstepswith

- Correct!
- The `with` keyword passes input parameters to an action when using `uses`:

```yaml
steps:
  - name: Set up Node.js
    uses: actions/setup-node@v4
    with:
      node-version: '20'
      cache: 'npm'
```

|                | `with`                    | `env`                    |
| -------------- | ------------------------- | ------------------------ |
| **Purpose**    | Inputs to an action       | Environment variables    |
| **Used with**  | `uses:` (actions)         | `uses:` or `run:` (both) |
| **Defined by** | The action's `action.yml` | You                      |

Q86: Which of the following GitHub Actions syntax is used to run multiple commands in a single step?

- Using && to chain commands❌
- Using a multiline string with | ✅
- Defining commands in an array❌
- Separating commands with a semicolon ;❌

💡 https://docs.github.com/en/enterprise-cloud@latest/actions/using-workflows/workflow-commands-for-github-actions#example-of-a-multiline-string

- Correct! The marked answer is right.
- The | (block scalar) YAML operator preserves newlines,
- Each line is executed as a separate command

```yaml
steps:
  - name: Build and test
    run: |
      npm ci
      npm run build
      npm test
```

Q89: Which of the following can be used to limit the number of concurrent jobs running in a GitHub Actions workflow?

- max-jobs
- concurrency✅
- parallelism
- limit

💡 https://docs.github.com/en/enterprise-cloud@latest/actions/using-jobs/using-concurrency

- Correct! The marked answer is right.
- The concurrency
- The concurrency keyword ensures only one workflow (or job) runs at a time for a given group, cancelling or queuing others:

```yaml
name: CI

on: push

concurrency:
  group: ${{ github.workflow }}-${{ github.ref }}
  cancel-in-progress: true

jobs:
  build: ...
```
