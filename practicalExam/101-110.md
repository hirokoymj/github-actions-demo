Q101: How do you ensure that Upload Failure test report step is executed only if Run Tests step fails?

1.

```yaml
- name: Run Tests
  id: run-tests
  run: npm run test

- name: Upload Failure test report
  if: always() && steps.run-tests.outcome == 'failure'
  run: actions/upload-artifact@v3
  with:
    name: test-report
    path: test-reports.html
```

✅ Correct answer:

👉 Option 1

- name: Upload Failure test report
  if: always() && steps.run-tests.outcome == 'failure'

#### 💡 Why this is correct

🔑 Key behavior in GitHub Actions

- By default, steps after a failure are skipped
- To run a step even after failure, you must use:
- `always()`

#### ✔️ What Option 1 does

```yaml
if: always() && steps.run-tests.outcome == 'failure'
```

- always() → ensures the step runs even if previous step failed
- steps.run-tests.outcome == 'failure' → ensures it runs only when tests failed
- 👉 ✅ Perfect combination

Q102: Which context holds information about the event that triggered a workflow run?

💡 [doc](https://docs.github.com/en/actions/using-workflows/triggering-a-workflow#using-event-information)

✅ Correct answer: 👉 github.event

#### 💡 Why this is correct

- The github.event context contains all details about the triggering event
- It includes payload data such as:
  - PR info
  - commit data
  - issue details

#### 🧩 Example

```yaml
steps:
  - run: echo "${{ github.event.pull_request.title }}"
```

👉 Accesses the PR title from the event payload

#### 🧠 Key takeaway

`Event data → github.event`

Q103: In GitHub Actions, if you define both branches and paths filter, what is the effect on the workflow execution?

💡 [doc](https://docs.github.com/en/actions/using-workflows/workflow-syntax-for-github-actions#onpull_requestpull_request_targetbranchesbranches-ignore)

✅ Correct answer: 👉 the workflow will only run when both branches and paths are satisfied

#### 💡 Why this is correct

When you define both branches and paths filters:

- GitHub uses AND logic
- The workflow runs only if BOTH conditions match

#### 🧩 Example

```yaml
on:
  push:
    branches:
      - main
    paths:
      - 'src/**'
```

- 👉 This workflow runs only when:
- ✅ The push is to main AND
- ✅ Files under src/ are changed

#### 🧠 Key takeaway

`branches + paths → AND condition`

Q105: Which of the following statements accurately describes the behavior of workflow jobs referencing an environment's protection rules?

💡 [doc](https://docs.github.com/en/actions/deployment/targeting-different-environments/using-environments-for-deployment)

✅ Correct answer: 👉 workflow jobs won't start until all the environment's protection rules pass

#### 💡 Why this is correct

When a job targets an environment with protection rules (e.g., required reviewers, wait timers):

- ⏸️ The job is paused before starting
- 🔐 It waits until all protection rules are satisfied
- ▶️ Only then does the job begin execution

🧩 Example

```yaml
jobs:
  deploy:
    runs-on: ubuntu-latest
    environment: production # 👈 has protection rules
```

Q106: What is the purpose of the restore-keys parameter in actions/cache in GitHub Actions?

- specify the location of the cached files
- enable cross-OS cache functionality
- provide alternative keys to use in case of a cache miss
- indicate whether a cache hit occurred

💡 [doc](https://docs.github.com/en/actions/using-workflows/caching-dependencies-to-speed-up-workflows#managing-caches)

✅ Correct answer: 👉 provide alternative keys to use in case of a cache miss

#### 🧩 Example

```yaml
- uses: actions/cache@v4
  with:
    path: node_modules
    key: npm-${{ hashFiles('package-lock.json') }}
    restore-keys: |
      npm-
```

👉 Behavior:

1. Try exact match → npm-<hash>
2. If not found → try fallback → npm-

Q107: Which variable would you set to true in order to enable step debug logging?

- ACTIONS_STEP_DEBUG
- ACTIONS_WORKFLOW_DEBUG
- ACTIONS_RUNNER_DEBUG
- ACTIONS_JOB_DEBUG

💡 https://docs.github.com/en/actions/monitoring-and-troubleshooting-workflows/enabling-debug-logging

Q109: What is the purpose of the timeout-minutes keyword in a step?

💡 [doc](https://docs.github.com/en/actions/using-workflows/workflow-syntax-for-github-actions#jobsjob_idstepstimeout-minutes)

✅ Correct answer: 👉 it limits the execution time for individual step

#### 💡 Why this is correct

- timeout-minutes at the step level:
  - ⏱️ Limits how long that specific step can run
  - If exceeded → the step (and job) fails

#### 🧩 Example

```yaml
steps:
  - name: Long task
    run: sleep 600
    timeout-minutes: 5 # 👈 step will be stopped after 5 minutes
```
