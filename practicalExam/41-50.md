### Q41: When should you use caching?

- When you want to reuse files that do change often between jobs or workflow runs, such as build dependencies from a package management system.

- When you want to save binaries produced by a build job to use in a subsequent deploy job to deploy a new version of an application

- When you want to save files produced by a job to view after a workflow run has ended, such as built binaries or build logs.

- When you want to reuse files that don't change often between jobs or workflow runs, such as build dependencies from a package management system.

💡 https://docs.github.com/en/actions/using-workflows/caching-dependencies-to-speed-up-workflows#comparing-artifacts-and-dependency-caching

✅ Correct Answer:

```
- When you want to reuse files that do change often between jobs or workflow runs, such as build dependencies from a package management system.

- When you want to save binaries produced by a build job to use in a subsequent deploy job to deploy a new version of an application

- When you want to save files produced by a job to view after a workflow run has ended, such as built binaries or build logs.

- When you want to reuse files that don't change often between jobs or workflow runs, such as build dependencies from a package management system. ✅
```

- **Why this is correct**
  - Caching is designed for files that are downloaded/installed repeatedly but rarely change — like node_modules, Maven packages, pip packages, etc.

```yaml
- name: Cache node modules
  uses: actions/cache@v3
  with:
    path: ~/.npm
    key: ${{ runner.os }}-node-${{ hashFiles('package-lock.json') }}
```

- Every time your workflow runs, instead of downloading all packages again, it restores from cache → faster builds ⚡
- "Files that do change often": ❌ Caching frequently changing files defeats the purpose — cache would constantly miss
- "Binaries for deploy job" : ❌ That's artifacts
- "Save files to view after workflow ends": ❌ That's also artifacts
- **Key takeaway**
  - Cache = same files reused across multiple workflow runs (dependencies)
  - Artifact = files passed between jobs or saved for download after a run

### Q42: When should you use artifacts? (Select two.)

- Use artifacts to reuse files that don't change often between jobs or workflow runs, such as build dependencies from a package management system.

- Use artifacts to save binaries produced by a build job to use in a subsequent deploy job to deploy a new version of an application

- Use artifacts to create new versions of your application together with release notes, mentions and/or contributors

- Use artifacts to save files produced by a job to view after a workflow run has ended, such as test results or build logs.

💡 https://docs.github.com/en/actions/using-workflows/storing-workflow-data-as-artifacts#about-workflow-artifacts

✅ Correct Answers (two):

```
- Use artifacts to reuse files that don't change often between jobs or workflow runs, such as build dependencies from a package management system.

- Use artifacts to save binaries produced by a build job to use in a subsequent deploy job to deploy a new version of an application ✅

- Use artifacts to create new versions of your application together with release notes, mentions and/or contributors

- Use artifacts to save files produced by a job to view after a workflow run has ended, such as test results or build logs. ✅
```

- **Answer 1 — Passing files between jobs:**

```yaml
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - run: npm run build
      - uses: actions/upload-artifact@v3
        with:
          name: build-output
          path: ./dist

  deploy:
    needs: build
    runs-on: ubuntu-latest
    steps:
      - uses: actions/download-artifact@v3
        with:
          name: build-output
      - run: ./deploy.sh # uses the binary from build job
```

- **Answer 2 — Saving files for post-run inspection:**

```yaml
steps:
  - run: npm test
  - uses: actions/upload-artifact@v3
    with:
      name: test-results
      path: ./test-results.xml # viewable in GitHub UI after run
```

- "Reuse files that don't change often": ❌ That's caching
- "Create new versions with release notes" - ❌ That's a GitHub Release

### Q43: If a workflow runs on a feature-a branch, can it restore caches created in the default main branch?

- Yes, all branches can restore caches created on the default branch

- No, caches can only be restored from the same branch

- Yes, all caches can be accessed by workflows on any branch within the same repository

- Yes but only if no files were changed on feature-a branch

💡 https://docs.github.com/en/actions/using-workflows/caching-dependencies-to-speed-up-workflows#restrictions-for-accessing-a-cache

✅ Correct Answer:

```
- Yes, all branches can restore caches created on the default branch ✅
- No, caches can only be restored from the same branch
- Yes, all caches can be accessed by workflows on any branch within the same repository
- Yes but only if no files were changed on feature-a branch
```

- **How GitHub cache access works**
  - GitHub has a specific cache access hierarchy:

```
main (default branch) cache
        ↓ accessible by
   feature-a branch ✅
   feature-b branch ✅
   any other branch ✅
```

- The default branch cache acts as a shared fallback for all branches in the repository.

### Q44: To access an artifact that was created in another, previously triggered workflow run you can:

- Use the `actions/download-artifact` action with elevated permissions.
- Use the `actions/download-artifact` action and make sure the artifact is not expired
- Use the `actions/upload-artifact` action.
- You cannot access artifacts that were created in a different workflow run

💡 https://github.com/actions/download-artifact?tab=readme-ov-file#download-artifacts-from-other-workflow-runs-or-repositories

✅ Correct Answer:

```
- Use the `actions/download-artifact` action with elevated permissions.
- Use the `actions/download-artifact` action and make sure the artifact is not expired ✅
- Use the `actions/upload-artifact` action.
- You cannot access artifacts that were created in a different workflow run
```

- **Why this is correct**
  - `actions/download-artifact` supports downloading artifacts from other workflow runs using the run-id parameter:

```yaml
steps:
  - uses: actions/download-artifact@v4
    with:
      name: my-artifact
      run-id: 1234567890 # ← specific run ID from a previous workflow run
      github-token: ${{ secrets.GITHUB_TOKEN }} # ← required for cross-run access
```

### Q45: What should you use to store coverage reports or screenshots generated during a workflow that runs automated testing for a repository?

- Releases
- Artifacts
- Caches
- Packages

💡 https://docs.github.com/en/actions/using-workflows/storing-workflow-data-as-artifacts#comparing-artifacts-and-dependency-caching

✅ Correct Answer:

```
- Releases
- Artifacts ✅
- Caches
- Packages
```

- **Why Artifacts are the right choice**
- Coverage reports and screenshots are files produced by a job that you want to review after the workflow run ends — that's exactly what artifacts are designed for.

```yaml
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - run: npm test

      - uses: actions/upload-artifact@v4
        with:
          name: coverage-report
          path: ./coverage/

      - uses: actions/upload-artifact@v4
        with:
          name: screenshots
          path: ./test-screenshots/
```

**After the run, you can view and download them directly in the GitHub Actions UI**

| Need                                     | Use          |
| ---------------------------------------- | ------------ |
| Save test results / screenshots / logs   | ✅ Artifacts |
| Speed up builds by reusing dependencies  | ✅ Cache     |
| Publish versioned app binaries for users | ✅ Releases  |
| Host npm / Docker / Maven packages       | ✅ Packages  |

### Q46: You can only upload a single file at a time when using actions/upload-artifact action

- Only directories can be uploaded, not individual files
- False
- True

💡 https://docs.github.com/en/actions/using-workflows/storing-workflow-data-as-artifacts#uploading-build-and-test-artifacts

✅ Correct Answer:

```
- Only directories can be uploaded, not individual files
- False✅
- True
```

- You can upload single files, multiple files, directories, and wildcards all at once:

```yaml
# ✅ Single file
- uses: actions/upload-artifact@v4
  with:
    name: single-file
    path: ./output/result.txt

# ✅ Multiple files (wildcard)
- uses: actions/upload-artifact@v4
  with:
    name: test-results
    path: ./results/*.xml

# ✅ Entire directory
- uses: actions/upload-artifact@v4
  with:
    name: coverage
    path: ./coverage/

# ✅ Multiple paths
- uses: actions/upload-artifact@v4
  with:
    name: mixed-files
    path: |
      ./dist/
      ./logs/error.log
      ./reports/*.html
```

### Q47: In job deploy, if you want to access binaries (containing your application) that were created in job build you should

- upload the binaries as artifacts in deploy and download them in build
- cache the binaries in deploy and read the files from cache in build
- upload the binaries as artifacts in build and download them in deploy
- cache the binaries in build and read the files from cache in deploy

💡 https://docs.github.com/en/actions/using-workflows/storing-workflow-data-as-artifacts#comparing-artifacts-and-dependency-caching

✅ Your answer is correct!

```
- upload the binaries as artifacts in deploy and download them in build
- cache the binaries in deploy and read the files from cache in build
- upload the binaries as artifacts in build and download them in deploy ✅
- cache the binaries in build and read the files from cache in deploy
```

- **Why this is correct**
- Jobs run on separate, isolated runners — they cannot directly share files. Artifacts are the bridge between them.

```yaml
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - run: npm run build # creates binaries in ./dist

      - uses: actions/upload-artifact@v4 # ← UPLOAD in build
        with:
          name: app-binaries
          path: ./dist/

  deploy:
    needs: build # waits for build to finish
    runs-on: ubuntu-latest
    steps:
      - uses: actions/download-artifact@v4 # ← DOWNLOAD in deploy
        with:
          name: app-binaries

      - run: ./deploy.sh # uses the downloaded binaries
```

### Q50: Secrets and configuration variables can be scoped to: (Select three.)

- Multiple repositories that do not share an organization/enterprise
- An environment shared across multiple repositories
- A single repository
- The entire organization, or selected repositories in an organization
- A specific job in a workflow
- An environment in a repository
- A specific workflow in a repository

💡 https://docs.github.com/en/actions/using-workflows/sharing-workflows-secrets-and-runners-with-your-organization#sharing-secrets-and-variables-within-an-organization

✅ Correct Answer:

```
- Multiple repositories that do not share an organization/enterprise
- An environment shared across multiple repositories ❌
- A single repository✅
- The entire organization, or selected repositories in an organization✅
- A specific job in a workflow
- An environment in a repository✅
- A specific workflow in a repository
```

- ❌ "An environment shared across multiple repositories" — This doesn't exist in GitHub Actions. Environments are always scoped to a single repository. They cannot be shared across repositories.
- ✅ "An environment in a repository" is the correct option — you can define environment-level secrets/variables (e.g., production, staging) within a specific repo.

- 1️⃣ Repository-level secret: 👉 Secret stored in: Repo → Settings → Secrets → Actions
- 2️⃣ Organization-level secret👉 Secret stored in: Org → Settings → Secrets → Actions
  (can allow all or selected repos)
- 3️⃣ Environment-level secret: 👉 Secret stored in: Repo → Settings → Environments → (e.g. prod) → Secrets

### Summary

- `Cache`: Save and reuse files that don't change often — so you don't re-download them every run.
- `GITHUB_TOKEN`
  - **What it is:** Auto-generated, short-lived secret created at the start of every workflow run. Authenticates with the GitHub API — no manual setup needed.
  - **Permission values:** `read`, `write`, `none`
  - **Where to set:** Workflow level ✅ Job level ✅ Step level ❌
  - **Reusable workflows:** Permissions only flow **downward** — called workflow can restrict but never elevate.
  - **When to pass it explicitly:** Only when the action needs to reach outside the current run (e.g., cross-run artifact download).

  ```yaml
  github-token: ${{ secrets.GITHUB_TOKEN }}
  ```

  - **Only required when the action needs to make an authenticated API call to GitHub.**

| Action                                  | Need `GITHUB_TOKEN`? | Why                                                                    |
| --------------------------------------- | -------------------- | ---------------------------------------------------------------------- |
| `actions/download-artifact` (cross-run) | ✅ Yes               | Fetching artifacts from a _different_ workflow run requires API access |
| `actions/upload-artifact`               | ❌ No                | GitHub handles this internally during the current run                  |
| `actions/checkout`                      | ❌ Usually no        | Works without it for basic checkout                                    |

- `Secrets`
  - `${{ secrets.SECRET_NAME }}`
  - 3 scope levels:

  ```yaml
  # 1. Repository secret
  env:
    API_KEY: ${{ secrets.API_KEY }}

  # 2. Organization secret
  env:
    API_KEY: ${{ secrets.API_KEY }}

  # 3. Environment secret
  jobs:
    deploy:
      environment: production  # ← must declare the environment
      steps:
        - run: echo ${{ secrets.API_KEY }}
  ```

  - 1️⃣ Repository-level secret: 👉 Secret stored in: Repo → Settings → Secrets → Actions
  - 2️⃣ Organization-level secret👉 Secret stored in: Org → Settings → Secrets → Actions
    (can allow all or selected repos)
  - 3️⃣ Environment-level secret: 👉 Secret stored in: Repo → Settings → Environments → (e.g. prod) → Secrets
