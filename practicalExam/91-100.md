Q91: How can you specify the operating system for a job in GitHub Actions?

- Using the os keyword
- Using the platform keyword
- Using the env keyword
- Using the `runs-on` keyword ✅

💡 [doc](https://docs.github.com/en/actions/using-workflows/workflow-syntax-for-github-actions#jobsjob_idruns-on)

✅ Correct answer: `runs-on` keyword

```yaml
name: CI Pipeline
on: [push]
jobs:
  build:
    runs-on: ubuntu-latest # ← specifies the OS/runner
    steps:
      - uses: actions/checkout@v4
      - name: Run a script
        run: echo "Running on Ubuntu!"
```

```yaml
jobs:
  test:
    runs-on: ${{ matrix.os }}
    strategy:
      matrix:
        os: [ubuntu-latest, windows-latest, macos-latest]

    steps:
      - uses: actions/checkout@v4
      - run: echo "Testing on ${{ matrix.os }}"
```

Q97: How does the actions/cache action in GitHub Actions handle a cache miss?

- by terminating the workflow if a cache miss occurs
- by automatically creating a new cache if the job is completed successfully ✅
- by requiring manual intervention to create a new cache
- by searching for a cache in other repositories

💡 [doc](https://docs.github.com/en/actions/using-workflows/caching-dependencies-to-speed-up-workflows#managing-caches)

✅ Correct answer:

- The whole point of actions/cache is to be self-healing — on first run (or after package-lock.json changes) it builds the cache automatically for all future runs.

```yaml
name: Cache Demo

on: [push]

jobs:
  build:
    runs-on: ubuntu-latest

    steps:
      - uses: actions/checkout@v4

      - name: Cache node modules
        uses: actions/cache@v4
        with:
          path: ~/.npm
          key: ${{ runner.os }}-node-${{ hashFiles('**/package-lock.json') }}
          restore-keys: |
            ${{ runner.os }}-node-

      - run: npm install
      - run: npm test
```

Q98: How can you specify the schedule of a GitHub actions workflow to run on weekdays only?

- it is not possible in GitHub actions
- 👉use the on: schedule: cron event trigger
- add a condition in the workflow YAML for weekdays
- use the on: schedule: weekdays event trigger

💡 [doc](https://docs.github.com/en/actions/using-workflows/events-that-trigger-workflows#schedule)

```yaml
name: Weekdays Only

on:
  schedule:
    - cron: '0 9 * * 1-5' # runs at 9:00 AM UTC, Monday–Friday

jobs:
  build:
    runs-on: ubuntu-latest

    steps:
      - uses: actions/checkout@v4
      - name: Run weekday task
        run: echo "Today is a weekday!"


┌─ minute        (0–59)
│  ┌─ hour       (0–23)
│  │  ┌─ day of month  (1–31)
│  │  │  ┌─ month      (1–12)
│  │  │  │  ┌─ day of week   (0–6, Sun=0, Mon=1 ... Fri=5, Sat=6)
│  │  │  │  │
0  9  *  *  1-5
```

- "M H D M W"
- zooming out from small → large time units:
- Minute → Hour → Day → Month → Weekday

| Cron          | Read as                         |
| ------------- | ------------------------------- |
| `0 9 * * *`   | At 9:00 AM, every day           |
| `0 9 * * 1-5` | At 9:00 AM, Mon–Fri             |
| `0 0 1 * *`   | At midnight, 1st of every month |
| `*/5 * * * *` | Every 5 minutes                 |

Q99: What is the recommended approach for storing secrets larger than 48 KB?

- 👉 encrypt and store secrets in the repository but keep the decryption passphrase as a secret
- store large secrets directly as repository secrets to avoid limitations
- secrets larger than 48 KB cannot be stored
- avoid storing large secrets entirely to ensure security

💡 [doc](https://docs.github.com/en/actions/security-guides/using-secrets-in-github-actions#limits-for-secrets)

```yaml
name: Large Secret Demo
on: [push]
jobs:
  decrypt:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Decrypt large secret file
        run: |
          gpg --quiet --batch --yes \
              --passphrase "${{ secrets.LARGE_SECRET_PASSPHRASE }}" \
              --output secret.json \
              secret.json.gpg       # ← encrypted file stored in repo
```

- The key idea: GitHub Secrets stores the passphrase, the repo stores the encrypted file. Neither alone is useful without the other.

Q100: Select status check functions in GitHub Actions

- status(), always(), cancelled() and failure()
- 👉 success(), always(), cancelled() and failure()
- completed(), always(), cancelled() and failure()
- state(), always(), cancelled() and failure()

💡 [doc](https://docs.github.com/en/actions/learn-github-actions/expressions#status-check-functions)

- The correct answer is — success(), always(), cancelled(), and failure().

```yaml
name: Status Check Functions Demo

on: [push]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - run: npm test

      - name: Runs only when previous steps succeeded
        if: ${{ success() }}
        run: echo "All good!"

      - name: Runs no matter what
        if: ${{ always() }}
        run: echo "Always runs!"

      - name: Runs when workflow was cancelled
        if: ${{ cancelled() }}
        run: echo "Workflow was cancelled!"

      - name: Runs when previous steps failed
        if: ${{ failure() }}
        run: echo "Something went wrong!"
```
