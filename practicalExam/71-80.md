### Q71: How can you override an organization-level GitHub Secret API_KEY with a different value when working within a repository? (Select two.)

- By creating a environment secret with the name ENVIRONMENT_API_KEY
- By creating a repository secret with the name OVERRIDE_API_KEY
- By creating a environment secret with the name OVERRIDE_API_KEY
- By creating a environment secret with the same name API_KEY ✅
- By creating a enterprise secret with the name OVERRIDE_API_KEY
- By creating a repository secret with the same name API_KEY ✅ .
- By creating a enterprise secret with the same name API_KEY ❌
- By creating a repository secret with the name REPOSITORY_API_KEY

💡 https://docs.github.com/en/actions/reference/security/secrets

```yaml
Enterprise: API_KEY = "ent-value"    ← lowest, used as last resort
Organization: API_KEY = "org-value"    ← overrides enterprise
Repository: API_KEY = "repo-value"   ← overrides org + enterprise ✅
Environment: API_KEY = "env-value"    ← overrides all ✅
```

- Why enterprise secret is wrong: Enterprise secret has LOWEST precedence

### Q72: What components can be reused within a GitHub Organization? (Select four.)

- Workflow Templates ✅
- Artifacts ❌.
- Configuration Variables❌✅
- Cache
- Environment Variables ✅
- Self Hosted Runners ❌✅
- Secrets ✅

- **Answer**
  - Workflow Templates ✅
  - Secrets ✅
  - Configuration Variables ✅
  - Self-Hosted Runners ✅

```yaml
Organization
  ├── Workflow Templates    → stored in org's .github repo
  │     └── repos pick them when creating new workflows
  │
  ├── Secrets              → org settings → secrets
  │     └── accessible in all permitted repos as secrets.MY_SECRET
  │
  ├── Configuration Vars   → org settings → variables
  │     └── accessible in all repos as vars.MY_VAR
  │
  └── Self-Hosted Runners  → org settings → runners
        └── any repo can use runs-on: [self-hosted, linux]
```

- Artifacts❌ - Scoped to a single workflow run only
- Cache❌ - Scoped to a single repository only
- Environment Variables❌ - Defined inside workflow files, not org-level

### Q73: How many jobs will be executed in the following workflow?

```yaml
jobs:
  matrix-job:
    runs-on: ubuntu-latest
    strategy:
      matrix:
        pet: [cat, dog]
        color: [pink, brown]
        include:
          - color: white
            pet: dog
    steps:
      - run: echo "Hello ${{ matrix.color }} ${{ matrix.pet }}"
```

- 6 ❌
- 5 ✅
- 4
- 7

💡 https://docs.github.com/en/actions/using-jobs/using-a-matrix-for-your-jobs#using-a-matrix-strategy

- Total: 4 + 1 = 5 jobs
- Base combinations (pet × color = 2 × 2 = 4 jobs):
- cat-pink, cat-brown, dog-pink, dog-brown, dog-white
- `include entry: { color: white, pet: dog }`
- The key rule: an include entry only expands an existing combination

### Q75: In a workflow that has multiple jobs, all running on GitHub-hosted runners, is it true that all jobs are guaranteed to run on the same runner machine?

- Only if they use the same runs-on label ✅❌
- Yes
- No ✅
- Only if they run in parallel

💡 https://docs.github.com/en/actions/using-jobs/choosing-the-runner-for-a-job#choosing-github-hosted-runners

- The right answer is No.
- GitHub-hosted runners provision a fresh, isolated virtual machine for each job, every time. Even if two jobs both specify `runs-on: ubuntu-latest`, there is no guarantee they land on the same physical machine —

### Q80: What is the correct syntax for setting the directory for all run commands in a workflow?

- set working-directory under job❌

```yaml
defaults:
  run:
    shell: bash
job:
  working-directory: ./scripts
```

- set directory under job❌

```yaml
defaults:
  run:
    shell: bash
job:
  directory: ./scripts
```

- set directory under defaults.run❌

```yaml
defaults:
  run:
    shell: bash
    directory: ./scripts
```

- set working-directory under defaults.run. ✅

```yaml
defaults:
  run:
    shell: bash
    working-directory: ./scripts
```

💡 https://docs.github.com/en/actions/using-workflows/workflow-syntax-for-github-actions#defaultsrunworking-directory

- Correct! The marked answer is right.
- The working-directory key must be nested under defaults.run, alongside shell:
- **A few things worth knowing about defaults.run:**
  - It accepts exactly two keys: `shell` and `working-directory` — nothing else

```yaml
jobs:
  my-job:
    runs-on: ubuntu-latest
    defaults:
      run:
        working-directory: ./scripts # only applies to this job
    steps:
      - run: echo "I'm running in ./scripts"
```
